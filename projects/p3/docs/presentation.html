결제 · 수강 · 환불 워크플로우

무료 강의

결제 단계 없이 initiatePayment가 즉시 Enrollment 생성.
이미 환불(REFUNDED)된 수강이면 재활성화.

유료 강의

PENDING 결제 생성 → 토스 위젯 → 승인 성공 시 COMPLETED + 수강 등록 동시 처리.

진도 30% 초과 시 환불 불가.
환불 시 결제 REFUNDED + 수강 회수, 재구매 허용.

05

강사(INSTRUCTOR) 학습자료 관리

강의 단어장

강의에 연결된 단어장 생성 — 단어·뜻·예문 CRUD.
공개 시 수강생이 연습에 사용, 비공개는 강사만. 삭제·토글은 확인 다이얼로그로 보호.

스피킹 프롬프트

스피킹 연습용 문장 + 힌트 등록. 강의 연결 또는 전체 공통.
활성/비활성 토글로 학생 노출 제어. 하드코딩을 DB화.

학생에게 노출되는 규칙 (권한·스코핑)

• 단어장: 학생은 수강 중(ACTIVE)인 강의의 공개 단어장만 연습 가능 — 본인 소유 또는 공개+수강 조건으로 접근 검증.
• 스피킹 프롬프트: 수강 중 강의 + 전체 공통(courseId=null)만 노출 — 무관한 강의 프롬프트 유출 차단.
• 공유 단어장의 SM-2 스케줄은 본인 소유 단어장에만 저장해 학생 간 진도 오염 방지.

07

구조화 로깅 & 인앱 알림

구조화 로깅 (SystemLog)

• logger.info/warn/error(category, message, { userId, meta })
• 카테고리: AUTH · PAYMENT · ENROLLMENT · ADMIN · CRON · SYSTEM
• 로깅 실패가 본 기능을 막지 않도록 try/catch 격리

인앱 알림 (Notification)

• 도메인 이벤트 → 알림 생성: QNA_ANSWERED, PAYMENT_DONE, COURSE_APPROVED/REJECTED, INSTRUCTOR_APPROVED 등
• 헤더 NotificationBell + /my/notifications

왜 분리했나

로그는 운영자가 사후에 추적하는 기록(상태 전이·실패 원인), 알림은 사용자에게 즉시 전달하는 이벤트. 결제 승인/실패처럼 하나의 사건이 양쪽 모두를 남긴다 — 관리자는 로그로 추적하고, 사용자는 알림으로 결과를 받는다.

08

관리자 모니터링 & Cron 배치

모니터링 대시보드

/admin/monitoring — 일별 통계(DailyStat) 추이.
/admin/logs — 레벨·카테고리 필터로 시스템 로그 열람.
/admin/payments — 결제 현황.

Cron Route Handlers

• daily-stats — 일별 지표 집계 적재
• weekly-reminders — 진도 정체 학생에게 학습 리마인드 메일
• cleanup-tokens — 만료 이메일 토큰 정리

보호 & 실행

모든 cron 엔드포인트는 Authorization: Bearer ${CRON_SECRET} 검증 후 동작 — 외부에서 임의 호출 차단. 외부 스케줄러(EventBridge/crontab 등)가 주기적으로 호출하는 구조로, 배치 로직을 앱과 한 코드베이스에서 관리.

멀티스테이지 Dockerfile

• deps → builder → runner 3단계, Next output: standalone로 런너 슬림화
• NEXT_PUBLIC_*는 빌드 타임 인라인이라 build arg로 주입
• prisma CLI 의존성 문제로 마이그레이션은 builder 스테이지에서 별도 수행

EC2 배포 흐름

• 로컬에서 이미지 빌드 → docker save로 전송(EC2 RAM 부족 회피)
• EC2: docker load → docker compose up -d --no-build
• .env의 AUTH_URL을 공개 주소로(로그인·메일 링크 기준), 열린 포트(80)에 매핑

환경 분리

DB URL은 DATABASE_URL 환경변수 우선(도커=postgres:5432, 로컬=localhost:5434). 메일은 SMTP_* 미설정 시 MailHog, 설정 시 실제 SMTP(인증 포함)로 자동 전환.

End-to-End 시나리오 점검

① 유료 수강 결제

강의 상세 → 결제 → 토스 위젯(테스트 카드) → 승인 → 자동 수강 등록 → 학습 시작. success 새로고침해도 멱등 처리로 안전.

② 강사 자료 → 학생 학습

강사가 공개 단어장 + 스피킹 프롬프트 등록 → 수강생이 /my/practice에서 플래시카드·퀴즈·스피킹으로 연습 → 정답률 누적.

③ 운영 관측

결제·수강 발생 → SystemLog 적재 + 사용자 알림 → 관리자 모니터링/로그/결제 화면에서 확인 → 주간 리마인드 메일 발송.

④ 환불 가드

진도 30% 이하만 환불 가능 — 초과 시 거절 사유 표시. 환불 후 재구매 가능, 결제는 REFUNDED로 보존.

12

회고 & 정리

잘 된 점

• 결제 승인을 멱등 + 트랜잭션으로 설계해 새로고침·중복요청에도 안전.
• 로그(추적)와 알림(전달)을 분리해 운영 관측과 사용자 경험을 동시에 확보.
• standalone + 멀티스테이지로 배포 가능한 컨테이너 완성.

아쉬운 점 / 트레이드오프

• 결제는 토스 테스트 모드 — 실거래 정산/세금계산서는 범위 외.
• 환불은 DB 상태 처리까지(테스트 키라 실제 취소 API 미호출).
• 공유 단어장 SM-2는 학생별 진도 테이블 대신 소유 단어장만 영속화.

다음 단계

실 결제 계약 + 환불/정산 API, 학생별 단어장 진도 테이블(VocabularyProgress), 도메인 + HTTPS(Nginx 리버스 프록시), 외부 스케줄러 연결, 그리고 Sentry 등 외부 모니터링 연동.
